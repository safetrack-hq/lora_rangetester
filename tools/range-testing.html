Link Budget Calculator

Predicts the theoretical free-space range from Tx power, antenna gains, and the receiver sensitivity for the chosen spreading factor. Free-space is an optimistic ceiling: ground reflection, foliage, and terrain cut it sharply.

SX1262 max +22 dBm
SX1262 datasheet Table 3-8, 125 kHz, RX Boosted
headroom for fading; 10–20 typical
FSPL(dB) = 20·log₁₀(dkm) + 20·log₁₀(fMHz) + 32.44. Max path loss = Ptx + Gt + Gr − loss − fade − sensitivity. Range solved from FSPL.

Fresnel Zone Calculator

Even with line-of-sight, the link needs clearance through the 1st Fresnel zone, and beyond the radio horizon the earth's curvature blocks it entirely. Use this to pick antenna heights.

F₁ at midpoint (m) = 8.657·√(dkm/fGHz); 60% clearance is the practical minimum. Radio horizon (4/3-earth) dkm = 4.12·(√h₁ + √h₂), h in m.

Airtime Calculator

Range Testing Overview

Two identical radios. The requester (nice!nano) sends a REQ_LOG on a button press; the responder (tx_test) replies with a LocationPacket carrying its GPS fix plus the RSSI/SNR it measured on the request. The requester logs both ends' positions and both halves of the link budget to its microSD card. Drive or walk the nodes apart until the link breaks. The farthest distance still delivering good packets is the measured range.

Roles

NodeJob
Requester / Logger
(nicenano, NODE_ID=1)		On button press, sends REQ_LOG addressed to the responder. On receiving a LocationPacket, measures return-path RSSI/SNR, snapshots its own GPS, and writes one CSV row. Missing packet_ids = losses.
Responder / Beacon
(tx_test, NODE_ID=2)		Continuously in RX. On a REQ_LOG whose targetId matches, measures forward-path RSSI/SNR, snapshots its GPS, builds a LocationPacket, transmits.

Role is fixed at build time by the PlatformIO env (nicenano vs tx_test) and the NODE_ID/TARGET_ID build flags in platformio.ini.

Why GPS lives in the payload

Range = distance between the two antennas at the instant of the transaction. The responder puts its lat/lng inside the LocationPacket, so a single RX log file is self-sufficient: every row already has both endpoints (RX GPS from the local module, TX GPS from the payload). No clock-syncing two logs, no post-hoc GPX join.

The log processor also accepts a legacy V1 single-GPS CSV plus a fixed reference point so old logs still load.

Packet Format

Wire packets (request / response)

V2 is a two-step transaction, not a one-way beacon. The requester sends a 6-byte REQ_LOG; the responder replies with a 23-byte LocationPacket carrying its GPS fix and the RSSI/SNR it measured on the request.

struct Header {          // 6 bytes — prefix of every packet
  uint8_t  magic;       // 0x56 (SFTRK_MAGIC)
  uint8_t  flag;        // sftrk_flag_t: 0x30 REQ_LOG · 0x40 ACK · 0x10/0x20 GPS_VALID/INVALID
  uint16_t nodeId;      // sender
  uint16_t targetId;    // intended recipient
};
struct AckPacket {       // 6 bytes (Header only) — carries REQ_LOG / ACK
  struct Header header;
};
struct LocationPacket {  // 23 bytes — responder → requester
  struct Header header;
  int16_t  rx_rssi;       // / 10  (dBm×10, responder's RX of REQ_LOG)
  int16_t  rx_snr;        // / 10  (dB×10)
  bool     gps_fix;
  int32_t  gps_lng;       // × 1e7
  int32_t  gps_lat;       // × 1e7
  uint8_t  gps_numSat;
  uint8_t  gps_avgGsvSnr; // reserved (0)
  uint16_t packet_id;     // monotonic from responder
};
All structs __attribute__((packed)), little-endian. lat/lng as int32 ×1e7 ≈ 1.1 cm resolution, no float on the wire. See docs/PACKET_FORMAT_V2.md for full details.

RX CSV row (logger → SD)

Header written once per file. One row per received LocationPacket. RSSI/SNR stored ×10 (int) to avoid float on the MCU; the processor divides by 10 for display.

event_type,event_time,event_pps_micros,node_id,target_id,packet_id,
rx_late7,rx_lnge7,rx_sats,         // logger GPS (requester)
tx_late7,tx_lnge7,tx_sats,         // responder GPS (from LocationPacket)
rx_rssix10,rx_snrx10,              // requester's RX of LocationPacket (return path)
tx_rssix10,tx_snrx10,              // responder's RX of REQ_LOG (forward path)
packet_len,                        // 23
latency_us                         // round-trip µs (REQ_LOG TX_DONE → LocationPacket RX_DONE)
Filenames N{node}_{seq}.csv (e.g. N01_0019.csv), sequential, never overwritten. See docs/PACKET_FORMAT_V2.md for full spec.
Loss / PDR. The log records only received packets. Packet delivery ratio at a given distance = received packet_ids ÷ expected span (max−min+1). Gaps in packet_id are the dropped packets — the processor reports them per distance bin.

Field Procedure

1
Pre-flight (bench). Confirm both nodes get GPS fix (sat count > 4), TX is beaconing, RX is writing rows to a fresh CSV. Note SF/BW/CR and Tx power — they decide sensitivity and time-on-air.
2
Plan the route. Use the link-budget calc for an expected range, and Fresnel / horizon to sanity-check line-of-sight and antenna height for that distance.
3
Place the base. Park one node at a known, open, elevated spot. Record its exact coordinate (it's also in every payload, but write it down).
4
Rove. Move the other node outward along the route. Keep antennas vertical and clear of the body/vehicle. Pause at markers so each distance gets several packets.
5
Find the edge. Continue until RSSI approaches sensitivity and packets start dropping. The last distance with acceptable PDR is the range for that config.
6
Recover & process. Power down cleanly (so the SD flush commits), pull the card, drop the CSV into the log processor for the range curve + map.
Watch RSSI vs. SNR. LoRa decodes below the noise floor — SNR can be negative on good packets. A packet that drops with RSSI still well above sensitivity points to interference or a Fresnel/obstruction problem, not raw range.
