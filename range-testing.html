Range Testing Overview

Two identical radios. One transmits GPS-stamped beacon packets on a fixed cadence; the other receives them, measures signal quality (RSSI / SNR), and logs both ends' positions to its microSD card. Drive or walk the nodes apart until the link breaks. The farthest distance still delivering good packets is the measured range.

Roles

NodeJob
TX (beacon)Every T seconds, sample its GPS, build a packet containing its own lat/lng + a monotonic packet_id, transmit.
RX (logger)On each received packet: read RSSI & SNR, sample its own GPS, parse the TX's GPS out of the payload, write one CSV row. Missing packet_ids = losses.

Both ends run the same firmware; NODE_ID + a runtime role flag pick TX vs RX. Either node can be the static "base"; the other roves.

Why GPS lives in the payload

Range = distance between the two antennas at the instant of the packet. Putting the TX position inside the payload means a single RX log file is self-sufficient: every row already has both endpoints (RX GPS from the local module, TX GPS from the payload). No clock-syncing two logs, no post-hoc GPX join.

FreeRTOS tasks: gpsPoll (TinyGPS++ decode → shared latestFix under mutex), lora_handleRx (DIO1 ISR → radioIrqQueue → read packet + RSSI/SNR), writeLogTask (drains logQueue → 512 B buffer → SD, periodic flush() sync).

SX1262 and SD share one SPI bus: bring the radio up then sleep() it (NSS high → MISO Hi-Z) before SD I/O.

Firmware gap (intended vs current). The payload-GPS design above requires the TX coordinates to be parsed from the packet and written as extra CSV columns (tx_lat_e7, tx_lng_e7). Current src/main.cpp defines a single-GPS header and its logQueue consumer isn't wired yet — the log processor below therefore also accepts a single-GPS CSV plus a fixed "reference point" so you can test the tooling before the firmware catches up.

Packet Format

Beacon payload (TX → air)

Compact fixed layout — carries TX position and GPS-PPS-synced timestamp so the RX side can compute latency.

struct Beacon {        // 22 bytes
  uint8_t  node_id;    // who sent it
  uint32_t packet_id;  // monotonic, for loss detection
  int32_t  lat_e7;     // TX latitude  × 1e7
  int32_t  lng_e7;     // TX longitude × 1e7
  uint64_t tx_gps_us;  // TX GPS-synced Unix time, µs (since epoch)
  uint8_t  flags;      // fix-valid, sats bucket, ...
};
lat/lng as int32 ×1e7 ≈ 1.1 cm resolution, no float on the wire. See docs/PACKET_FORMAT.md for full details.

RX CSV row (logger → SD)

Header written once per file. One row per received packet. GPS-PPS-synced timestamps enable latency measurement.

time_valid,utc_iso,event_type,node_id,packet_id,
rx_lat_e7,rx_lng_e7,          // local GPS (logger)
tx_lat_e7,tx_lng_e7,          // from payload (beacon)
gps_fix_time_us,rssi_dbm,snr_db,
tx_gps_us,rx_gps_us           // GPS-synced Unix µs — latency = (rx − tx) / 1000
Filenames N{node}_{seq}.csv (e.g. N02_0007.csv), sequential, never overwritten. See docs/PACKET_FORMAT.md for full spec.
Loss / PDR. The log records only received packets. Packet delivery ratio at a given distance = received packet_ids ÷ expected span (max−min+1). Gaps in packet_id are the dropped packets — the processor reports them per distance bin.

Field Procedure

1
Pre-flight (bench). Confirm both nodes get GPS fix (sat count > 4), TX is beaconing, RX is writing rows to a fresh CSV. Note SF/BW/CR and Tx power — they decide sensitivity and time-on-air.
2
Plan the route. Use the link-budget calc for an expected range, and Fresnel / horizon to sanity-check line-of-sight and antenna height for that distance.
3
Place the base. Park one node at a known, open, elevated spot. Record its exact coordinate (it's also in every payload, but write it down).
4
Rove. Move the other node outward along the route. Keep antennas vertical and clear of the body/vehicle. Pause at markers so each distance gets several packets.
5
Find the edge. Continue until RSSI approaches sensitivity and packets start dropping. The last distance with acceptable PDR is the range for that config.
6
Recover & process. Power down cleanly (so the SD flush commits), pull the card, drop the CSV into the log processor for the range curve + map.
Watch RSSI vs. SNR. LoRa decodes below the noise floor — SNR can be negative on good packets. A packet that drops with RSSI still well above sensitivity points to interference or a Fresnel/obstruction problem, not raw range.

Link Budget Calculator

Predicts the theoretical free-space range from Tx power, antenna gains, and the receiver sensitivity for the chosen spreading factor. Free-space is an optimistic ceiling: ground reflection, foliage, and terrain cut it sharply.

SX1262 max +22 dBm
SX1262 datasheet Table 3-8, 125 kHz, RX Boosted
headroom for fading; 10–20 typical
FSPL(dB) = 20·log₁₀(dkm) + 20·log₁₀(fMHz) + 32.44. Max path loss = Ptx + Gt + Gr − loss − fade − sensitivity. Range solved from FSPL.

Fresnel Zone Calculator

Even with line-of-sight, the link needs clearance through the 1st Fresnel zone, and beyond the radio horizon the earth's curvature blocks it entirely. Use this to pick antenna heights.

F₁ at midpoint (m) = 8.657·√(dkm/fGHz); 60% clearance is the practical minimum. Radio horizon (4/3-earth) dkm = 4.12·(√h₁ + √h₂), h in m.

Airtime Calculator

Airtime sets your beacon cadence and, on duty-limited bands, your legal max. Bigger SF = more range but longer airtime = fewer packets/sec and coarser distance sampling.

Semtech LoRa airtime model, SX1262 preamble (n+6.25 symbols — the SX126x modem adds 2 symbols vs SX127x). Note: 915 MHz (FCC) has no fixed duty cycle but a 400 ms dwell-time limit on frequency-hopping; airtime still bounds packets/sec.
